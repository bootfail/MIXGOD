# Phase 01.1: SoundCloud/YouTube Import - Research

**Researched:** 2026-03-18
**Domain:** yt-dlp integration, .NET Process spawning, React import UI
**Confidence:** HIGH

## Summary

This phase adds URL-based audio import from YouTube and SoundCloud. The core technical challenge is spawning yt-dlp as a server-side process from the .NET backend, streaming progress to the frontend, and feeding downloaded files into the existing analysis pipeline. The existing codebase already has a well-established pattern for this: `AnalysisService.cs` spawns Python processes via `System.Diagnostics.Process`, and `AnalysisQueueProcessor.cs` manages background jobs via `Channel<T>`. The download service follows the exact same pattern.

yt-dlp is the only tool needed -- it handles YouTube, SoundCloud, metadata extraction (title, artist, thumbnail, duration), and audio format conversion in a single binary. No additional libraries are required on the server. On the client, the existing Zustand store pattern, polling infrastructure, and neon-themed component library provide everything needed for the import dialog.

**Primary recommendation:** Create a `DownloadService` that wraps yt-dlp process spawning (mirroring `AnalysisService`), a `DownloadQueueProcessor` background job (mirroring `AnalysisQueueProcessor`), and a new `POST /api/tracks/import` endpoint. On the client, build an `ImportDialog` component and extend the existing Track type with source metadata fields.

### Pattern 1: yt-dlp Two-Phase Execution
**What:** First call yt-dlp --dump-json to get metadata (fast, no download), then call yt-dlp to download audio.
**When to use:** Always -- allows immediate UI feedback with title/thumbnail before download completes.
**Example:**
```csharp
// Phase 1: Metadata extraction (fast, ~1-2s)
// yt-dlp --dump-json --no-download "URL"
// Returns JSON with: title, uploader, thumbnail, duration, id, webpage_url

// Phase 2: Audio download (slow, progress-tracked)
// yt-dlp -x --audio-format mp3 --audio-quality 0 --newline
//   --progress-template "download:%(progress._percent_str)s|%(progress._speed_str)s|%(progress._eta_str)s"
//   --write-thumbnail -o "%(id)s.%(ext)s" "URL"
```

### Pattern 2: Download-Then-Analyze Pipeline
**What:** Download completes -> file stored -> track updated -> job queued for analysis (same Channel<AnalysisJob>).
**When to use:** Every URL import follows this flow.
**Example:**
```
User pastes URL
  -> POST /api/tracks/import { urls: [...] }
  -> For each URL:
     1. Create Track with DownloadStatus=Queued, AnalysisStatus=Pending
     2. Queue DownloadJob to Channel<DownloadJob>
  -> Return track IDs immediately

DownloadQueueProcessor picks up job:
  1. yt-dlp --dump-json -> extract metadata -> update Track (title, artist, thumbnail)
  2. yt-dlp -x -> download audio -> update Track (filePath, DownloadStatus=Done)
  3. Queue AnalysisJob to existing Channel<AnalysisJob>

AnalysisQueueProcessor picks up job (existing, unchanged):
  1. Python analyzer -> BPM, key, energy, genre
  2. Peak generation -> waveform data
  3. Update Track (AnalysisStatus=Done)
```

### Pattern 3: Progress Tracking via Polling
**What:** Extend existing status endpoint to include download progress. Client polls same endpoint.
**When to use:** Reuse the analysisStore polling pattern for download status.
**Example:**
```csharp
// Extend GET /api/tracks/{id}/status response:
{
  "id": "abc123",
  "downloadStatus": "downloading",  // pending | downloading | done | error
  "downloadProgress": 65,           // 0-100 percentage
  "downloadEta": "30s",
  "analysisStatus": "queued",       // queued | analyzing | done | error
  "errorMessage": null
}
```

### Pattern 4: URL Validation (Client-Side)
**What:** Regex-based URL validation for YouTube and SoundCloud URLs before sending to server.
**When to use:** In ImportDialog, validate as user types/pastes.
**Example:**
```typescript
const YOUTUBE_REGEX = /^https?:\/\/(www\.)?(youtube\.com\/watch\?v=|youtu\.be\/|youtube\.com\/shorts\/)[\w-]+/;
const SOUNDCLOUD_REGEX = /^https?:\/\/(www\.)?soundcloud\.com\/[\w-]+\/[\w-]+/;

function validateUrl(url: string): 'youtube' | 'soundcloud' | null {
  if (YOUTUBE_REGEX.test(url)) return 'youtube';
  if (SOUNDCLOUD_REGEX.test(url)) return 'soundcloud';
  return null;
}
```

### Anti-Patterns to Avoid
- **Downloading in the request thread:** Never run yt-dlp synchronously in the controller action. Always queue to background job and return immediately.
- **Parsing yt-dlp default progress output:** Default output uses carriage returns and overwrites lines. Always use `--newline` + `--progress-template` for parseable output.
- **Storing download progress in Track model directly:** Use a separate ConcurrentDictionary for volatile progress data (percentage/ETA changes every second). Track model stores final state only.
- **Re-implementing metadata parsing:** yt-dlp --dump-json returns rich metadata. Use it as the primary source, then apply title-splitting regex on top.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Audio downloading from YouTube/SC | Custom HTTP downloader | yt-dlp | Handles DRM, age gates, rate limiting, format selection, 1000+ edge cases |
| Audio format conversion | NAudio/ffmpeg wrapper | yt-dlp -x --audio-format | yt-dlp already wraps ffmpeg for post-processing |
| Thumbnail downloading | Separate HTTP fetch | yt-dlp --write-thumbnail | Downloads best quality thumbnail automatically |
| Metadata extraction | HTML scraping | yt-dlp --dump-json | Returns structured JSON with title, uploader, duration, thumbnail URL |
| Title/Artist splitting | Nothing -- but don't over-engineer | Simple regex + common suffix stripping | "Artist - Title (Official Video)" covers 80% of cases; expose manual edit for the rest |
| URL validation | Full URL parser | Regex for YouTube + SoundCloud patterns | Only two platforms; regex is sufficient and fast |

**Key insight:** yt-dlp does 90% of the heavy lifting. The backend is just a thin orchestration layer: receive URL, spawn yt-dlp, parse output, store files, queue analysis.

## Common Pitfalls

### Pitfall 1: yt-dlp Not Installed
**What goes wrong:** Process spawn fails with "file not found" or similar OS error.
**Why it happens:** yt-dlp is a system dependency, not a NuGet package.
**How to avoid:** On startup, check `yt-dlp --version` and log a clear error. In the import endpoint, return a 503 with user-friendly message if yt-dlp is not available.
**Warning signs:** FileNotFoundException or Win32Exception on Process.Start().

### Pitfall 2: yt-dlp Output Encoding on Windows
**What goes wrong:** Unicode characters in video titles cause garbled output or JSON parse failures.
**Why it happens:** Windows console defaults to cp1252/cp437, not UTF-8.
**How to avoid:** Set `ProcessStartInfo.StandardOutputEncoding = Encoding.UTF8` and `StandardErrorEncoding = Encoding.UTF8`. Also pass `--encoding utf-8` to yt-dlp if needed.
**Warning signs:** Corrupted characters in track titles, JSON deserialization exceptions.

### Pitfall 3: Long-Running Downloads Blocking the Queue
**What goes wrong:** A single slow download (large file, slow connection) blocks other downloads.
**Why it happens:** Sequential processing without concurrency control.
**How to avoid:** Use SemaphoreSlim(2) like AnalysisQueueProcessor uses SemaphoreSlim(3). Allow 2 concurrent downloads to balance throughput vs. bandwidth.
**Warning signs:** Multiple queued downloads, first one taking minutes while others wait.

### Pitfall 4: yt-dlp Progress Output Parsing Failures
**What goes wrong:** Progress regex fails to match, percentage shows as "N/A" or unparseable.
**Why it happens:** yt-dlp output varies by site, format, and version. Some downloads skip progress (e.g., DASH segments show per-fragment progress).
**How to avoid:** Use `--progress-template` for structured output. Parse with fallback: if percentage is unparseable, show "Downloading..." without percentage. Never crash on parse failure.
**Warning signs:** "N/A" in progress fields, regex match returning null.

### Pitfall 5: Duplicate File Names in Storage
**What goes wrong:** Two downloads produce files with the same name, overwriting each other.
**Why it happens:** Using video title as filename (titles can collide).
**How to avoid:** Use trackId as the directory name (already the pattern in AudioStorageService). Store as `audio-storage/{trackId}/audio.mp3` and `audio-storage/{trackId}/thumb.jpg`.
**Warning signs:** Missing audio files after multiple downloads.

### Pitfall 6: SoundCloud Private/Geo-Restricted Tracks
**What goes wrong:** yt-dlp fails with "This track is not available" or similar.
**Why it happens:** SoundCloud has private tracks, geo-restrictions, and Go+ exclusive content.
**How to avoid:** Catch yt-dlp non-zero exit codes, parse stderr for known error patterns, and return meaningful error messages to the user ("Track is private", "Region restricted", "Requires SoundCloud Go+").
**Warning signs:** Generic "download failed" errors without actionable information.

```csharp
{
    public async Task<DownloadMetadata> GetMetadataAsync(string url, CancellationToken ct)
    {
        var psi = new ProcessStartInfo
        {
            FileName = "yt-dlp",
            Arguments = $"--dump-json --no-download \"{url}\"",
            RedirectStandardOutput = true,
            RedirectStandardError = true,
            UseShellExecute = false,
            CreateNoWindow = true,
            StandardOutputEncoding = Encoding.UTF8,
            StandardErrorEncoding = Encoding.UTF8
        };
        // ... spawn, read stdout JSON, parse title/uploader/thumbnail/duration
    }

    public async Task<string> DownloadAudioAsync(
        string url, string outputDir, Action<int, string>? onProgress, CancellationToken ct)
    {
        var psi = new ProcessStartInfo
        {
            FileName = "yt-dlp",
            Arguments = string.Join(" ",
                "-x", "--audio-format", "mp3", "--audio-quality", "0",
                "--newline",
                "--progress-template", "\"download:PROGRESS|%(progress._percent_str)s|%(progress._speed_str)s|%(progress._eta_str)s\"",
                "--write-thumbnail",
                "-o", $"\"{Path.Combine(outputDir, "audio.%(ext)s")}\"",
                $"\"{url}\""),
            RedirectStandardOutput = true,
            RedirectStandardError = true,
            UseShellExecute = false,
            CreateNoWindow = true,
            StandardOutputEncoding = Encoding.UTF8
        };
        // ... spawn, read stdout line-by-line, parse PROGRESS lines, invoke onProgress callback
    }
}
```

### Title Parsing Regex (C#)
```csharp
// Strip common video suffixes, then split "Artist - Title"
private static readonly Regex VideoSuffixes = new(
    @"\s*[\(\[](Official\s*(Video|Audio|Music\s*Video|Lyric\s*Video)|" +
    @"Lyric\s*Video|Music\s*Video|HD|HQ|4K|Visualizer|" +
    @"Audio|Lyrics?|Official)[\)\]]\s*",
    RegexOptions.IgnoreCase | RegexOptions.Compiled);

private static readonly Regex ArtistTitleSplit = new(
    @"^(?<artist>.+?)\s*[-\u2013\u2014]\s*(?<title>.+)$",
    RegexOptions.Compiled);

public static (string title, string artist) ParseTitle(string rawTitle)
{
    var cleaned = VideoSuffixes.Replace(rawTitle, "").Trim();
    var match = ArtistTitleSplit.Match(cleaned);
    if (match.Success)
        return (match.Groups["title"].Value.Trim(), match.Groups["artist"].Value.Trim());
    return (cleaned, string.Empty);
}
```

## Open Questions

1. **ffmpeg availability on dev machine**
   - What we know: yt-dlp requires ffmpeg for --extract-audio conversion. The PeakService uses audiowaveform which may or may not have been installed with ffmpeg.
   - What's unclear: Whether ffmpeg is already on PATH on the developer's Windows machine.
   - Recommendation: Add a startup health check that verifies both `yt-dlp --version` and `ffmpeg -version`. If missing, log installation instructions (`pip install yt-dlp` and `winget install ffmpeg`).

2. **Download progress storage granularity**
   - What we know: Progress changes every ~1 second during download. Track model uses ConcurrentDictionary.
   - What's unclear: Whether to store volatile progress in the Track model or a separate in-memory dictionary.
   - Recommendation: Use a separate `ConcurrentDictionary<string, DownloadProgress>` for volatile progress data. Track model stores only final states. Status endpoint merges both.

3. **Thumbnail format and size**
   - What we know: YouTube thumbnails are typically 1280x720 JPG. SoundCloud artwork varies.
   - What's unclear: Whether to resize/compress or store as-is.
   - Recommendation: Store original thumbnail as-is (yt-dlp --write-thumbnail). Serve via a new `/api/tracks/{id}/thumbnail` endpoint. Let the browser handle display sizing. Thumbnails are small (50-200KB) so no optimization needed for Phase 01.1.
