<objective>
Build the .NET backend API that handles audio file upload, storage, and the analysis orchestration pipeline. Tracks are uploaded via multipart POST, stored to disk, and automatically queued for Python Essentia analysis and waveform peak generation.

Purpose: This is the server-side backbone. Without it, the frontend cannot upload files, trigger analysis, or retrieve results. The background queue with bounded concurrency ensures batch uploads of 30+ tracks don't overwhelm the server.

Output: Working API endpoints for track upload, status polling, audio streaming, and peak data retrieval. Background analysis queue that calls the Python analyzer from Plan 01.
</objective>

  <behavior>
    - Writing a job to the analysis channel triggers background processing
    - Analysis queue processes max 3 tracks concurrently (SemaphoreSlim)
    - AnalysisService calls `python analyzer.py <filepath>` and parses JSON stdout
    - PeakService calls `audiowaveform` CLI or falls back to NAudio peak extraction
    - Track status updates: queued -> analyzing -> done (or error on failure)
    - Failed analysis sets status to 'error' with error message, does not crash queue
  </behavior>
  <action>
    1. Create BackgroundJobs/AnalysisQueueProcessor.cs as IHostedService:
       - Inject Channel<AnalysisJob> (unbounded channel, created as singleton in DI)
       - AnalysisJob record: TrackId, FilePath
       - ExecuteAsync: read from channel, use SemaphoreSlim(3) for bounded concurrency
       - For each job: update track status to "analyzing", call AnalysisService, call PeakService, update track with results, set status to "done"
       - On failure: set status to "error", log exception, continue processing queue
       - Graceful shutdown via CancellationToken

    2. Create Services/AnalysisService.cs:
       - AnalyzeAsync(string filePath, CancellationToken ct) -> AnalysisResult
       - Starts Python process: `python src/analysis/analyzer.py <absolute_path>`
       - Reads stdout as JSON, deserializes to AnalysisResult
       - Timeout: 120 seconds per track (kill process if exceeded)
       - Returns parsed result or throws on failure
       - Configurable Python path via appsettings.json (default: "python")

    3. Create Services/PeakService.cs:
       - GeneratePeaksAsync(string audioFilePath, string outputPath, CancellationToken ct) -> string (output path)
       - Primary: call `audiowaveform -i <path> -o <output.json> --pixels-per-second 20 --bits 8`
       - Fallback (if audiowaveform not found): Use NAudio to read audio samples, downsample to peaks array (20 peaks/second), write JSON in wavesurfer.js format: `{"version":2,"channels":1,"sample_rate":44100,"samples_per_pixel":2205,"bits":8,"length":N,"data":[...]}`
       - Store peaks JSON alongside audio file in audio-storage/{trackId}/peaks.json

    4. Update Program.cs:
       - Register Channel<AnalysisJob> as singleton (Channel.CreateUnbounded<AnalysisJob>())
       - Register AnalysisQueueProcessor as hosted service
       - Register AnalysisService, PeakService as transient/scoped
       - Add configuration section for Python path and audiowaveform path

    5. Write tests (AnalysisServiceTests.cs):
       - test_parse_analyzer_json_output: given valid JSON string, parses to AnalysisResult
       - test_analysis_timeout_throws: verify timeout handling
       - test_peak_fallback_when_audiowaveform_missing: PeakService uses NAudio fallback
       - test_queue_processes_jobs_concurrently: enqueue 5 jobs, verify max 3 run at once (mock AnalysisService with delay)
       - test_failed_analysis_sets_error_status: mock AnalysisService to throw, verify track status becomes "error"
